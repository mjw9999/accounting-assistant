    智能记账理财助手 —— 毕业答辩 24 道专业问题极简口述速记卡（答辩老师最爱版）
========================================================================
【使用说明】
1. 本速记卡重新以【本科答辩老师】的评审视角进行重构。去除了所有背书式的套话。
2. 答辩老师最忌讳“绕弯子”和“背大篇课本理论”。他们最喜欢听到“简单粗暴、直戳本质”的干货。
3. 每一道题的回答精简在 2-3 句话（60-90字左右），语速沉稳，口语自然，能让老师瞬间捕捉到您的技术亮点。
4. 您可以直接将这些极简回答复刻粘贴到您的《答辩记录书》自选问题作答中。

1. 本项目采用了哪种开发模式？该开发模式有哪些特点？
答：项目采用“前后端分离”开发模式。前端 Vue 3，后端 Spring Boot 3.3，双方基于标准 JSON 格式进行 RESTful 异步通信。特点是前后端物理和逻辑彻底解耦，可并行开发，且页面由浏览器端渲染，极大地减轻了后端服务器的 CPU 运算压力。

2. 请谈谈你项目数据库表结构的整体设计思路。
答：设计思路是“分层清晰、轻物理约束、重业务隔离”。10张表划分为基础数据、核心账务投资、社区分享三个逻辑层。全表没有物理外键，通过 Java 业务和 Spring 事务保证强一致性，有效降低了高频写时的表锁阻塞。同时，在用户ID、记账日期等高频检索列建立了 B+ 树索引。

3. 项目数据表之间存在哪些关联关系？
答：主要是一对多和一对一的逻辑关联。用户与账单流水、投资记录是一对多；理财产品与投资明细是一对多；投资记录与赎回明细、心得分享是一对一。这些关系在表里通过 ID 进行逻辑关联，方便未来进行水平分库分表扩展。

4. 项目中是否使用了缓存技术？主要起到了什么作用？
答：目前没有引入缓存。因为理财和记账涉及真实账户资金结余，对数据“强一致性”要求极高，所以直接读写 MySQL 保证严谨事务。但在架构设计上已预留接口，未来高并发下，我们会用 Redis 缓存产品列表等更新慢的字典数据，分担数据库 90% 以上的检索压力。

5. 请说明本项目接口的定义规范以及前端调用方式。
答：接口严格遵循 RESTful 规范，用标准 HTTP 动词操作资源，统一返回 `ApiResponse` 格式的 JSON。前端用 Axios 异步调用，配置了全局拦截器：请求时自动在 Header 带上 Token 授信；响应时集中捕获 401 登录失效和 400 校验异常并拉起 Element Plus 气泡警示。

6. 项目开发中，你是如何对用户输入的数据进行合法性校验的？
答：采用前后端双重拦截校验。前端利用 Element Plus 的表单规则进行正则前置校验，不合规直接拦截；后端在 DTO 属性上声明 Validation 校验注解（如 `@NotNull`, `@DecimalMin`），Controller 接收时打上 `@Valid` 强校验，不合规由全局异常处理器拦截，拼接后友好返回前端。

7. 系统针对报错、超时、异常操作等场景做了哪些容错处理？
答：第一，全局异常捕获屏蔽，拦截所有系统级报错，统一对用户提示“系统异常”，隐藏敏感堆栈防范窥探；第二，记账与理财联动操作全面挂载 `@Transactional` 事务，中途出错自动完整回滚；第三，Axios 设置 10秒超时防卡死，并在后台提供一键对账与数据自愈接口（`/fix-data`）。

8. 项目中用到了哪些核心算法？简要说明算法原理及具体应用场景。
答：核心用到三个算法：一是用 `BigDecimal` 四舍五入保留两位小数的高精度收益折算算法，防御浮点数精度丢失；二是自研的 HmacSHA256 Token 数字签名算法，防止用户篡改 ID 和越权；三是手写恒定时间比对算法（Constant-Time Equals），让 Token 每次验证耗时完全相同，物理防御侧信道的时序攻击。

9. 项目从哪些方面采取措施，保障系统的数据安全与访问安全？
答：有四个安全防线：一是数据库存储安全，用户密码使用 BCrypt 算法加盐加密，防逆向破解；二是接口访问安全，集成 Spring Security 框架进行 USER 和 ADMIN 角色控制，防越权；三是 Token 用自研 HmacSHA256 签名防篡改；四是防 SQL 注入，JPA 底层采用预编译参数绑定机制。

10. 简述页面渲染的基本原理，以及你在项目中做过的页面加载优化。
    答：渲染基于 Vue 3 虚拟 DOM 的数据响应式驱动。状态改变时通过 Diff 算法比对新旧虚拟 DOM 差异，计算出最小修改路径 Patch 到真实 DOM。页面加载优化上：我们对 Element Plus 组件按需引入，通过 Vite 压缩剔除代码，并对 ECharts 图表做生命周期异步懒加载，提升秒开体验。

11. 结合你的项目，谈谈前后端分离架构的优点与不足。
    答：优点是开发解耦、支持前后端并行开发，且客户端渲染降低了后端主机的 CPU 压力。不足是带来了跨域（CORS）限制。我们的对策是：在开发阶段配置 Vite 本地代理（Proxy）转发 API 请求，生产部署阶段在后端配置 `@CrossOrigin` 跨域注解安全解决。

12. 本项目主要采用了哪些软件测试方式？各类测试的作用是什么？
    答：采用了三种测试：一是使用 JUnit 5 单元测试，专门测试核心收益公式与 Token 签名解析逻辑，确保最底层算法精准；二是接口集成测试，用 Postman 测试 Controller 接口路由、参数校验及异常返回；三是端到端测试，模拟真实用户进行极端值输入和防越权拦截，保证最终交付质量。

13. 项目开发过程中，你是如何使用版本控制工具进行代码管理的？
    答：使用 Git 版本控制，遵循经典的 Git Flow 分支模型。`master` 分支放稳定生产代码，`develop` 分支用于日常开发集成，开发新功能拉 `feature/*` 特性分支测试后再合并，修 Bug 拉 `bugfix/*` 分支。每次 commit 严格规范书写，保证代码轨迹清晰。

14. 项目技术选型阶段，为什么放弃了其他同类框架，选择当前技术栈？
    答：后端用 Spring Boot + JPA，开箱即用，JPA 相比 MyBatis 拼 SQL 更类型安全，规避手写 SQL 隐患。前端用 Vue 3 + Vite，Proxy 响应式性能远超 Vue 2，Vite 热更新极快。安全上摒弃庞大的第三方 JWT 库，用 JDK 原生 Mac 手写 Token，避免了外部包的供应链安全漏洞。

15. 系统在高并发访问场景下可能存在哪些问题？你是如何规避和优化的？
    答：面临的核心问题是并发写锁阻塞和频繁报表查询导致的数据库 CPU 飙升。我们的规避方案是：前瞻性地将大表升级为支持行锁与 MVCC 的 InnoDB 引擎，防范 MyISAM 表锁死锁；对高频条件建立复合索引；并实行细粒度事务控制，只读查询不挂载事务，缩短锁持有时间。

16. 请完整梳理本项目从需求分析、设计、开发、测试到收尾的全过程。
    答：系统经历了标准的软件生命周期：首先需求分析调研出记账乱、理财对账难的痛点，确立核心功能；其次概要设计规划了 MVC 架构和 10 张表的 E-R 模型；然后开发阶段前后端并行编码，后端实现强事务联动对账；接着在测试阶段排查越权漏洞；最后在收尾阶段通过 `/fix-data` 进行全量历史对账清洗，圆满交付。

17. 项目代码中如何实现异常的捕获、抛出与后续处理逻辑？
    答：采用声明式全局异常拦截架构。当业务层校验不通过（比如余额不足）时，手动抛出自定义 `BusinessException` 中断流程，触发事务回滚。在控制层最外层，利用 `@RestControllerAdvice` 结合 `@ExceptionHandler` 自动拦截业务及系统异常，统一包装为 `ApiResponse.fail()` 格式的 JSON，向前端隐藏敏感堆栈。

18. 项目界面是否做了多设备、多屏幕的适配兼容处理？
    答：做了完备的响应式适配。前端基于 Element Plus 提供的 24 栏响应式栅格系统，配合媒体查询动态调整布局。卡片组件在手机端独占 24 栏垂直单列展示，平板上占据 12 栏双列展示，电脑大屏上占 6 或 8 栏多列展示，侧边栏在手机端自动折叠为顶部抽屉，保证多端无损呈现。

19. 在本次毕业设计开发中，你是如何运用软件工程思想规范开发流程的？
    答：核心是树立规范化与健壮性的工程设计理念。一是在架构上实现高内聚低耦合，划分职责明晰的 MVC 三层；二是在开发中严格编写 RESTful 契约接口和 commit 日志；三是把资金和对账一致性作为软件健壮性核心，通过声明式事务和防御性强校验排除逻辑漏洞。

20. 若项目后期数据量大幅增长，可通过哪些方案优化数据查询与存储效率？
    答：有五个调优方案：一是存储引擎升级，大表全面 InnoDB 化释放行锁吞吐；二是建立复合索引覆盖常用查询；三是搭建 MySQL 主从复制，在应用层实现读写分离；四是水平分表，流水表按用户 ID 进行哈希分表或按时间分区；五是心得模糊检索接入 Elasticsearch 分布式搜索引擎，避免 LIKE 扫表。

21. 项目对接第三方接口过程中遇到过哪些问题？分别如何排查和解决？
    答：遇到过三类问题：一是接口超时与重复提交，我们通过在 HTTP 头附带业务幂等性 Token 配合后端分布式锁解决；二是批量 CSV 导入时数据格式异常导致报错，我们在 DTO 前置解析阶段进行兼容性清洗；三是实名变更导致的历史流水姓名滞后，我们开发了专门的 `/fix-data` 事务级一键对账同步端点进行覆写。

22. 结合你的系统，说明MVC架构的分层结构及每层在项目中的具体作用。
    答：系统严格遵循 MVC 模式：View 层是前端由 Vue 3 渲染的动态视图，负责收集用户交互和展示图表报表；Controller 层对应后端的 Controller 接口，负责 RESTful 路由映射、前置数据强校验与统一 JSON 包装返回；Model 层包含后端的 JPA 实体类 Domain 和 Repository 持久层，承载了底层的核心记账对账数据模型以及高精度折算算法。

23. 结合你的毕业设计课题，谈谈目前相关的前沿技术与研究方向。
    答：前沿方向主要有三个：一是利用 LLM 大语言模型和多模态图像识别，实现“说一段语音”或“拍小票图片”即可智能分析并零交互自动记账；二是智能投顾，通过机器学习对用户流水大数据画像，自动定制资产配置；三是基于区块链技术对财务流水进行司法级防篡改存证，让个人资产具备无可争议的公信力。

24. 如果让你重新迭代开发本项目，你会在哪些方面进行优化和改进？
    答：一是对残留的 MyISAM 大表全面升级为 InnoDB 以引入强事务和行级锁；二是在 Controller 与 JPA 之间挂载 Redis 高速缓存，并引入 RabbitMQ 消息队列，将联动的同步阻塞操作改造为异步消费，实现消峰解耦；三是深度重构控制器，将数据清洗和三表联动等复杂逻辑下沉到 Service 实现类中，瘦身 Controller。
